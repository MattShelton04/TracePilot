# Unified Store Error Handling - Comprehensive Implementation Plan

## Executive Summary

**Problem:** The 10+ Pinia stores in TracePilot have inconsistent error handling patterns with 200+ lines of repetitive try-catch boilerplate. Each store independently implements loading states, error conversion, request deduplication, and stale request prevention.

**Solution:** Extend the existing `useCachedFetch` composable to support all common store patterns, then systematically refactor each store to use it. This will eliminate ~200 lines of code, standardize error handling, and enable future enhancements like retry logic and telemetry.

**Impact:**
- **Code Reduction:** ~200 lines (15-20% of store code)
- **Maintainability:** Single source of truth for async state management
- **Feature Velocity:** Global features (retry, telemetry) require changes in one place
- **Consistency:** All stores handle errors the same way

### Key Problems

1. **Inconsistent Error Conversion:**
   - `String(e)` (sessions, launcher, worktrees, search) - loses stack traces
   - `e instanceof Error ? e.message : String(e)` (sessionDetail, useCachedFetch) - better but still duplicated
   - No structured error types

2. **Request Deduplication:**
   - `fetchPromise` tracking (sessions)
   - `inflight` Map (useCachedFetch)
   - `indexingPromise` (sessions)
   - No deduplication (launcher, sessionDetail per-section loaders)

3. **Stale Request Prevention:**
   - `requestToken` counter (sessionDetail) - good
   - `searchGeneration` counter (search) - good
   - `loadGeneration` counter (worktrees) - good
   - `generation` counter (useCachedFetch) - good
   - No protection (sessions.fetchSessions, launcher actions)

4. **Multiple Error States:**
   - sessionDetail has 8 error refs (main error + 7 per-section)
   - Other stores have single error ref
   - No consistent pattern for partial failures

## Proposed Solution

### Phase 1: Enhance `useCachedFetch` Composable

**File:** `/home/runner/work/TracePilot/TracePilot/apps/desktop/src/composables/useCachedFetch.ts`

#### New Features to Add:

1. **Multi-Fetcher Support** - For stores with multiple related endpoints (analytics pattern)
   ```typescript
   export interface MultiCachedFetchOptions<TFetchers> {
     fetchers: TFetchers;  // Record of fetcher functions
     cacheKeyFn?: (name: keyof TFetchers, params: any) => string;
   }

   export interface MultiCachedFetchResult<TFetchers> {
     fetch: (name: keyof TFetchers, params: any, options?: { force?: boolean }) => Promise<void>;
     // ... other methods
   }
   ```

2. **Simple Action Wrapper** - For actions that don't need caching (launcher pattern)
   ```typescript
   export interface ActionOptions<TParams, TResult> {
     action: (params: TParams) => Promise<TResult>;
     onSuccess?: (result: TResult) => void;
     onError?: (error: string) => void;
   }

   export interface ActionResult<TParams, TResult> {
     execute: (params: TParams) => Promise<TResult | null>;
     loading: Ref<boolean>;
     error: Ref<string | null>;
   }

   export function useAction<TParams, TResult>(options: ActionOptions<TParams, TResult>): ActionResult<TParams, TResult>
   ```

3. **Request Token Pattern** - For operations that need cancellation (sessionDetail pattern)
   ```typescript
   export interface CancellableOptions<TParams, TResult> {
     fetcher: (params: TParams, signal: AbortSignal) => Promise<TResult>;
     onCancel?: () => void;
   }

   export interface CancellableResult<TParams, TResult> {
     fetch: (params: TParams, options?: { force?: boolean }) => Promise<TResult | null>;
     cancel: () => void;
     generation: Readonly<Ref<number>>;
     // ... other properties
   }

   export function useCancellableFetch<TParams, TResult>(options: CancellableOptions<TParams, TResult>): CancellableResult<TParams, TResult>
   ```

4. **Per-Section Error Tracking** - For stores with multiple independent sections (sessionDetail pattern)
   ```typescript
   export interface SectionOptions<TSections extends string> {
     sections: readonly TSections[];
   }

   export interface SectionResult<TSections extends string, TData> {
     loadSection: (section: TSections, fetcher: () => Promise<TData>, options?: { force?: boolean }) => Promise<void>;
     sectionData: Record<TSections, Ref<TData | null>>;
     sectionError: Record<TSections, Ref<string | null>>;
     sectionLoading: Record<TSections, Ref<boolean>>;
     loaded: Readonly<Ref<Set<TSections>>>;
   }

   export function useSectionLoader<TSections extends string, TData>(options: SectionOptions<TSections>): SectionResult<TSections, TData>
   ```

#### Implementation Strategy:

Instead of creating separate composables, we'll:
1. Keep `useCachedFetch` as-is (it's already good)
2. Create complementary composables:
   - `useAsyncAction` - For simple async operations (launcher, worktrees actions)
   - `useCancellableAction` - For operations that need cancellation (sessionDetail, search)
   - `useSectionLoader` - For multi-section loading (sessionDetail only)

This preserves the single responsibility principle and keeps composables focused.

```typescript

  loading.value = true;
  error.value = null;
  fetchPromise = (async () => {

      sessions.value = await listSessions();
    } catch (e) {
```

```typescript

  fetcher: () => listSessions(),
});

  await sessionsFetcher.fetch(undefined);
  if (!sessionsFetcher.error.value) {
    sessions.value = sessionsFetcher.data.value ?? [];
  }

```

**Before (lines 129-160 - reindex with custom deduplication):**
```typescript
async function reindex() {
  if (indexingPromise) {
    try {
      await indexingPromise;
      sessions.value = await listSessions();
    } catch (e) {
      const msg = String(e);  // ❌ Loses error type
      if (msg !== "ALREADY_INDEXING") {
        error.value = msg;
      }
    }
    return;
  }
  indexing.value = true;
  error.value = null;
  try {
    indexingPromise = reindexSessions();
    await indexingPromise;
    sessions.value = await listSessions();
  } catch (e) {
    const msg = String(e);
    if (msg !== "ALREADY_INDEXING") {
      error.value = msg;
    }
  } finally {
    indexingPromise = null;
    indexing.value = false;
  }
}
```

**After:**
```typescript
const reindexAction = useAsyncAction({
  action: () => reindexSessions(),
  deduplicate: true,  // Built-in deduplication
});

async function reindex() {
  const success = await reindexAction.execute(undefined);
  if (success) {
    await sessionsFetcher.fetch(undefined, { force: true });
    if (!sessionsFetcher.error.value) {
      sessions.value = sessionsFetcher.data.value ?? [];
    }
  } else if (reindexAction.error.value !== "ALREADY_INDEXING") {
    error.value = reindexAction.error.value;
  }
}

// Expose combined loading/error state
const indexing = reindexAction.loading;
```

## Appendix A: Error Conversion Best Practice

Current inconsistency:
```typescript
// ❌ Bad - loses stack trace
error.value = String(e);

// ❌ Bad - inconsistent
error.value = e instanceof Error ? e.message : String(e);

// ❌ Bad - verbose
const msg = String(e);
if (msg !== "ALREADY_INDEXING") error.value = msg;
```

Proposed standard (in composables):
```typescript
// ✅ Good - consistent, preserves type info
function toErrorMessage(e: unknown): string {
  if (e instanceof Error) return e.message;
  if (typeof e === 'string') return e;
  return String(e);
}

// Usage in composables
error.value = toErrorMessage(e);
```

**Note:** This function already exists at `packages/ui/src/utils/formatters.ts:139-150`. We should import and use it!
