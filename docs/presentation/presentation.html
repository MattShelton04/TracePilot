Copilot CLI Internals

What actually happens when you run copilot

Where does Copilot CLI live?

~/.copilot/
├── config.json ← global config
├── mcp-config.json ← MCP servers
├── session-state/ ← persisted sessions
│   ├── {uuid-1}/
│   ├── {uuid-2}/
│   └── ...
├── pkg/universal/{ver}/ ← installed versions
│   ├── app.js
│   ├── definitions/ ← agent YAMLs
│   └── schemas/ ← event schemas
├── agents/ ← user custom agents
├── skills/
└── logs/

What this means

  • Everything is local, file-based, under one directory
  • Sessions persist as UUID-named folders
  • Agent definitions are editable YAML files — not hardcoded
  • Each installed version ships its own schema at schemas/session-events.schema.json
  • Auto-updates download to pkg/tmp/ then extract — the loader picks the newest version

Copilot CLI ships roughly 4 releases per week. Event types have grown significantly across versions — the schema is the best compatibility contract.

Session-state folders

Each session = one UUID directory. The folder name is the session ID — not derived from the repo or path.

~/.copilot/session-state/{UUID}/
├── ✅ always
├── ~70%
├── ~50%
├── ~30%
├── ✅ always
├── ~60%
├── files/ ✅ always
└── inuse.{PID}.lock ← active sessions only

Key points

  • The folder is just a UUID — repo/branch/cwd metadata lives inside workspace.yaml
  • One repo can have many sessions
  • events.jsonl is the main data source — accounts for the vast majority of storage
  • TracePilot detects active sessions via inuse.*.lock files with freshness checks

The events.jsonl format

An append-only, line-delimited JSON event log. Every action Copilot takes becomes an event.

{
  "type": "tool.execution_start",
  "data": {
    "toolCallId": "tc-1",
    "toolName": "read_file",
    "arguments": { "path": "/test/foo.rs" }
  },
  "id": "evt-5",
  "timestamp": "2026-03-10T07:14:52.100Z",
  "parentId": "evt-4"   ← links to the assistant message that requested this tool
}

Events form a tree via parentId

Each event's parentId points to the event that triggered it. TracePilot reconstructs conversation turns by walking this tree:

session.start (id: evt-1, parentId: null)
├── user.message (id: evt-2, parentId: evt-1)
│   └── assistant.turn_start (id: evt-3, parentId: evt-2)
│       └── assistant.message (id: evt-4, parentId: evt-3)  ← requests tool call tc-1
│           ├── tool.execution_start (id: evt-5, parentId: evt-4)
│           │   └── tool.execution_complete (id: evt-6, parentId: evt-5)
│           └── assistant.message (id: evt-7, parentId: evt-4)  ← follow-up after tool result
│               └── assistant.turn_end (id: evt-8, parentId: evt-7)
└── session.shutdown (id: evt-9, parentId: null)

Tool calls nest under the assistant message that requested them. Subagent calls create deeper branches. TracePilot pairs starts with completions in O(1) via a HashMap index.

Why this matters

These are just YAML files in ~/.copilot/pkg/universal/{ver}/definitions/. You can override the model — swap claude-haiku-4.5 for claude-opus-4.6 and suddenly your explore subagent is dramatically more capable.


Custom agents live at

  • ~/.copilot/agents/ — user-level (all projects)
  • .github/agents/ — repo-level

The model field cycles above to show how easy it is to swap — that one line change is the difference between a cheap quick answer and a deep analysis.

The Problem

What happens after Copilot finishes working?

Pain points in AI-first workflows

🕵️

"What did it actually do?"

Sessions can run for hours with dozens of tool calls. The terminal scrollback is gone or buried. Which files did it read? What commands ran? What failed silently? You have no record beyond what's in your head.

💸

"How much is this costing me?"

You're burning premium requests across multiple models — Opus for complex work, Haiku for subagents, Sonnet in between. There's no dashboard showing your actual spend, no way to compare wholesale API cost vs premium request pricing, and no trend data.

🔍

"I solved this exact problem last week"

It was in some session, in some conversation turn, maybe with a specific tool call. But which session? You can't search across them. The solution is buried in a JSONL file you'd have to parse manually.

📊

"Which model/approach actually works?"

Different models, different prompting strategies, different tool combinations. You're making these choices by gut feel. There's no data on success rates, token efficiency, or which tools get used most.

🐛

"Why did that session go sideways?"

Rate limits, context truncation, compaction losing important context, silent errors — these happen and you don't notice until the output is wrong. There's no incident tracking, no health scoring, no warnings surfaced.

👁️

"What is it doing right now?"

You kicked off a long session and switched tabs. Is it still running? Has it stalled? Is it burning tokens on something useless? The terminal only shows you what already happened — there's no live view of an active session's progress.

⚡

"Managing parallel sessions is painful"

You want to run 3 Copilot sessions on different features in separate worktrees, each with the right model and config. That's a lot of manual terminal setup, git commands, and YAML editing every time.

🔄

"Am I getting better at this?"

You use Copilot every day but there's no feedback loop. No way to see if your prompting improved, if switching models helped, or how your usage patterns changed over time. You're flying blind on your own growth.

Also: 4 shared TypeScript packages (@tracepilot/types, @tracepilot/ui, @tracepilot/client, @tracepilot/config) and an experimental apps/cli companion tool.

Two-phase indexing pipeline

Phase 1 — Session metadata + analytics (fast)

Discover UUIDs→ Check mtimes + size→ Parse workspace.yaml→ Parse events.jsonl→ Reconstruct turns→ Upsert to SQLite

Phase 2 — Full-text search content (background)

Re-parse events→ Extract 9 content types→ FTS5 upsert→ BM25 ranking ready

Design choices

  • Incremental — only reindexes sessions whose events.jsonl mtime or size changed
  • Per-session SAVEPOINT — one bad session can't poison the entire batch
  • Parsing outside transactions — CPU-heavy work doesn't hold the database lock
  • Non-queued concurrency — if already indexing, rejects immediately (no queue buildup)
  • Phase 2 is async — kicked off automatically after Phase 1 completes, won't block the UI

Turn reconstruction

The flat event stream needs to become structured conversation turns. This is the trickiest part of the pipeline.

State Machine

Opens a turn on user.message. Auto-creates one if assistant/tool events appear first (which happens with resumed sessions). Buffers session events between turns and attaches them to the right context.

O(1) Tool Matching

Indexed lookup via HashMap<tool_call_id, (turn_idx, call_idx)>. When a tool.execution_complete arrives, we find its matching start in constant time — no reverse scans.

Subagent Model Inference

Fixed-point propagation: walk child tool calls up to their parent subagent call, inferring which model was used. Capped iterations to guard against cyclic parent IDs in malformed data.

Forward Compatibility

Unknown event types → Unknown(String). Schema mismatches → raw JSON Other(Value). All payload fields use Option<T>. Diagnostics track failures without dropping data.
