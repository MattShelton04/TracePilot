## 8. Forward Compatibility Assessment

### Strengths
- `Unknown(String)` catch-all variant in `SessionEventType` gracefully handles new event types
- All data struct fields use `Option<T>` for missing-field tolerance
- `ParseDiagnostics` tracks unknown events and deserialization failures
- Unknown events don't reduce session health scores (Info severity, 0.0 deduction)
- Event envelope (`id`, `timestamp`, `parentId`, `type`, `data`) has never changed

### Risks
- ~4 releases/week with potential schema changes
- No formal deprecation process from GitHub
- `session.start.data.version` field is always `1` (unreliable for version detection)
- `additionalProperties: false` in schema means strict validation would break

## 9. Recommendations

### Suggested next steps

1. **Expand `SessionEventType` enum** to cover all schema-defined event types (reduces `Unknown` noise in diagnostics)
2. **Add `copilot_version` to index DB** for version-aware filtering in the desktop app
3. **Add typed data structs** for high-value observed events (e.g., `assistant.usage` for per-turn token tracking)
4. **Run this analysis periodically** when new Copilot CLI versions are installed
5. **Consider automated schema diffing** in CI to catch breaking changes early
