JWT Authentication Refactor — Summary

Refactored the authentication module from session-based auth to stateless JWT tokens, improving scalability and enabling multi-device support.

Changes Overview

This task migrated the entire authentication flow from server-side sessions to JSON Web Tokens. The implementation uses RS256 signing with rotating key pairs stored in the system keychain.

Key Decisions

  • Token strategy: Short-lived access tokens (15min) with long-lived refresh tokens (7d) stored in HTTP-only cookies
  • Key rotation: Automated via a cron job every 24h, with a 2-key overlap window for graceful migration
  • Backward compatibility: Legacy session endpoints remain active for 30 days with deprecation warnings

Risk Assessment

Low risk. All changes are backward-compatible. The migration path is well-defined with the 30-day deprecation window. No database schema changes required — tokens are stateless.

Potential Concerns

  1. Token size (~800 bytes) may impact header limits on older proxies
  2. Clock skew tolerance set to 30s — may need adjustment for distributed systems
  3. Refresh token revocation requires a small Redis blocklist (added)
